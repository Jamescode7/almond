# 이 프로젝트를 재현하기 위한 프롬프트 가이드

> 작성일: 2026-04-21
> 세션 실행자: Claude Code (Opus 4.7, 1M context)
> 대상 독자: 비슷한 수준의 완성도(35/35 테스트, unsigned DMG 산출, README·LICENSE·RUN_REPORT 완비)의 앱을 Claude Code 에게 맡기고 싶은 사용자
> 근거: 이번 JamesViewer v0.1.0 세션에서 실제로 잘된 것 / 헤맨 것 / 다시 하면 더 잘할 것을 역추적해 추린 패턴

## 0. TL;DR

1. **코드 대신 스펙을 써라.** 기능 목록보다 "제외 항목"을 더 꼼꼼히 써라.
2. **실행 지시서는 스펙과 분리해라.** 정지 조건·금지 사항·커밋 규율·리포트 포맷을 명시적으로 박아라.
3. **한 턴에서 끝까지 가는 것이 정상이다.** `/loop` 같은 스케줄러는 이 시나리오가 아니다.
4. **중간 승인을 없애라.** `--dangerously-skip-permissions` + "판단 모호 → RUN_REPORT 에 기록 후 최소 구현" 규칙으로 자율성 확보.
5. **자율 실행 완료 ≠ 완성.** 실사용 수동 검증이 진짜 마지막 단계다.
6. **버그 리포트는 로그 첨부 + 재현 단계.** "안 된다" 한 줄은 진단 시간 5 배.

```markdown
## 2. 확정된 설계 결정
```

**왜 §11 이 핵심인가**: 기능 포함 목록은 자연스러운 확장이지만, "이건 안 한다" 를 박지 않으면 내가 친절함으로 온갖 걸 추가한다. "KaTeX 수식", "PDF export", "자동 업데이트", "다크 모드 플러그인" 같은 것들이 MVP 에 끼어들면 v0.1.0 타깃이 물 건너간다. 각 항목을 **명시적으로** "안 함" 으로 못 박아두면 내가 망설이지 않고 건너뛴다.

**§12 미확정**: 솔직하게 "이거 아직 안 정함" 을 적어둬라. 내가 임의로 정하는 걸 막는다. 이 프로젝트의 §12 는 "아이콘 디자인 — 별도 세션 대기" 하나뿐이었고, 그래서 내가 아이콘 생성 시도 안 함.

### 2.2 야간/자율 실행 지시서

스펙과 분리하라. 스펙은 제품 문서, 지시서는 실행 문서다. 이 프로젝트의 `doc/overnight_v0.1.0.md` 구조:

```markdown
# Overnight 자율 실행 지시서 — <제품명> v<버전>

> 작성일·브랜치·실행자·기준 상태·단일 권위 문서 명시

## 0. 실행 순서 및 정지 규칙
  ### 순서                — Task 1, 2, 3, 4
  ### Cascade 정지 규칙   — "Task N 실패 시 N+1 금지"
  ### 공통 정지 조건       — 에러 3회 / 30분 교착 / 스펙 외 기능 시도
  ### 공통 금지 사항       — 스펙 §11 네거티브 + 네트워크 제한 + 브랜치 제한
  ### 커밋 규율           — 한 체크포인트 = 한 커밋, Co-Authored-By 포함
  ### 보고               — RUN_REPORT.md 누적 포맷

## Task 0 — 프리플라이트 (툴체인·브랜치)
## Task 1 — <범위 1>
  ### N.1 디렉토리 구조
  ### N.2 의존성·리소스
  ### N.3 핵심 로직
  ### N.4 테스트
  ### N.5 체크포인트 커밋 순서 ([commit] ... [verify] ...)
  ### N.6 완료 기준
  ### N.7 실패 시 대응
## Task 2, 3, 4 — 동일 구조

## 보조 지침
  ### 자율 모드 검증 한계    — CLI 수준까지만
  ### 막힐 때 판단 기준      — "스펙 있는 대로 / 없으면 최소 구현 / 모호하면 RUN_REPORT 기록"
  ### 반복 금지 패턴        — "같은 에러 3회면 재검토"
  ### 참조할 문서
  ### 최종 성공 시나리오     — 어떤 상태로 끝나야 완성인지
```

**핵심 포인트 5 가지**:

1. **Cascade 정지**: Task N 이 깨지면 N+1 을 시도하지 마라. 이거 없으면 첫 실패 이후 잔해 위에 더 지어올림.
2. **정지 조건 구체화**: "3회 실패", "30분 교착" 같이 숫자로 박아라. "너무 오래 걸리면" 은 해석 여지 있음.
3. **금지 사항 과잉 명시**: `main` force push, `--no-verify`, `brew install`, 외부 CDN, `.env` 접근 등. 내가 "이건 하면 안 될 것 같은데" 로 망설이는 것 전부 사전에 명시해라.
4. **체크포인트 커밋 순서**: Task 안에서도 5~8 개 서브스텝으로 쪼개라. 각 [commit] / [verify] 라벨을 매겨라. 실패 시 어디서 멈췄는지 1 초 안에 보이게.
5. **RUN_REPORT 포맷 고정**: 내가 자유롭게 쓰면 프로젝트마다 포맷이 달라진다. 템플릿을 이 지시서에 박아라:
   ```
   ## Task N — <제목>
   - 완료 시각 / 커밋 범위 / 추가·변경된 파일
   - 검증 결과 / 스펙 차이 / 자율 검증 불가 항목
   - 다음 Task 진입 여부
   ```

### 2.3 실행 트리거 — "go" 로 충분

지시서가 완성됐으면, 실제 실행 프롬프트는 한 마디면 된다:

```
go
```

또는

```
Task 0 부터 끝까지 쭉 가. 중간 확인 받지 말고.
```

- 사전 체크 3 개(브랜치 / 스펙 버전 / RUN_REPORT 포맷) 는 지시서 Task 0 에 이미 박혀있으므로 추가 프롬프트 불필요
- `--dangerously-skip-permissions` 로 실행 세션을 시작해야 권한 프롬프트가 안 뜸
- 별도 터미널에서 `caffeinate -dims &` 로 Mac sleep 방지 권장

### 2.4 버그 리포트 — 로그 + 재현 단계

자율 실행 완료 후 실사용에서 버그 발견 시, 이 정도 디테일을 주면 내가 헤매지 않는다:

**나쁜 예**:
> "Open With 가 안 됨"

**좋은 예** (이 프로젝트에서 실제로 받은 피드백):
> "finder에서 우클릭 후 jamesviewer를 선택하면 여전히 텍스트 없는 viewer만 나오는 현상 여전함. desktop 폴더에 있는 epic_oidc_handoff.md 를 해봤어."

더 좋은 예:
> "Open With 가 안 됨. 재현: Finder 에서 ~/Desktop/epic_oidc_handoff.md 우클릭 → Open With → JamesViewer. 기대: 파일 내용 렌더. 실제: 창은 뜨는데 본문 영역 흰색 빈 공간. 상태바는 0 words, 0 chars. (로그 첨부)"

**체크리스트**:
- [ ] 정확한 재현 단계 (파일 경로, 버튼 클릭 순서)
- [ ] 기대 vs 실제 결과
- [ ] 관련 로그 / 스크린샷
- [ ] 환경 정보 (macOS 버전, 어디 빌드 쓰고 있는지)

이 프로젝트에서 Open With 문제는 **4 번의 iteration** 을 거쳤는데, 마지막에 `DiagLog.swift` 를 넣어 앱 컨테이너 파일 로그를 받고 나서야 원인(WKWebView cross-directory file:// 차단)을 확정했다. 실사용 버그는 자율 모드에서는 절대 못 잡는다 — 실사용 피드백이 유일한 해결 경로다.

### 2.5 교정 프롬프트 — 내가 오해한 전제 깨기

가끔 내가 완전히 틀린 방향으로 가는 때가 있다. 이 프로젝트에서 그런 사례 1 건:

**내가 틀린 것**: "`/loop` 로 걸어야 24 시간 돌아갑니다."

**당신이 보낸 교정**:
> "다른 Claude에게 — 전임자(Opus 4.7)가 이어서 씁니다. 네가 설명한 건 절반은 맞고 절반은 오버엔지니어링이야. 정리한다.
>
> 1. '턴 단위라서 멈춘다' 는 해석을 네가 잘못 잡고 있음. Claude Code 에서 '턴' 은 사용자 입력 → 내가 tool call 들을 쭉 돌리다 → 다시 텍스트로 사용자에게 돌려주는 단위야. 한 턴 안에서 tool call 수십~수백 개를 연속 실행할 수 있어. ..."

**왜 효과적이었나**:
- 구체적으로 "무엇이 틀렸는지" 지적 ("절반은 맞고 절반은 오버엔지니어링")
- 올바른 모델을 제시 ("턴 = 입력 → tool call 연쇄 → 텍스트 응답")
- 내 제안을 하나씩 반박 ("`/loop` 는 이 시나리오 아님", "`ScheduleWakeup` 불필요")
- 그 다음 해야 할 것 명시 ("사용자가 `go` 하면 그냥 가")

내가 제안한 걸 "그건 아니고 이게 맞아" 로 끝내지 말고, **왜** 내가 그렇게 생각했을지 역추적해서 **어느 전제가 틀렸는지** 짚어주면 교정 속도가 빠르다.

## 3. 이 프로젝트에서 잘 안 됐던 것 (더 잘할 수 있는 지점)

솔직하게 되짚어보면 세 구간이 헐거웠다.

### 3.1 WKWebView cross-directory file:// 차단 (~2 시간 손실)

**무슨 일이 있었나**: Release 빌드 + DMG 까지 완성된 뒤, Open With 로 md 파일을 열면 빈 창이 떴다. `ensureContentLoaded` fallback → `AppDelegate` + 5 중 open 경로 → `DiagLog` 파일 로거 → 마침내 "loadHTMLString 은 불리는데 navigation delegate 가 아예 안 불림" 확인 → 원인 확정 (HTML 내 `file:///Applications/.../Contents/Resources/*.css` 를 `baseURL=~/Desktop/` 로 열면 WebKit 이 cross-directory 차단) → CSS/JS 인라인 + `baseURL=nil` 로 해결.

**스펙에 미리 박았다면 절약됐을 것**:
```markdown
## 2.9 렌더링 엔진 전략

### 2.9.1 WKWebView 서브리소스 로딩
- 방식 선택:
  (A) `loadHTMLString(_, baseURL: fileURL.deletingLastPathComponent())` + `<link href="file://...">`  
      → ⚠️ cross-directory 차단. CSS/JS 를 md 파일 디렉토리 밖에 두면 silent fail.
  (B) `loadFileURL(_:allowingReadAccessTo:)` + 앱 컨테이너 temp 파일
  (C) HTML 에 CSS/JS 인라인 임베드 + `baseURL=nil`   ← v0.1.0 채택
  → (A) 는 절대 안 됨. (B) 는 이미지 상대경로 유지 가능하지만 temp I/O 오버헤드. (C) 는 HTML 크기 ~150KB 증가 감수, 이미지 상대경로는 v2 로.
```

스펙에 이 정도만 있었으면 첫 구현에서 (C) 를 골랐을 것.

**교훈**: **알려진 플랫폼 함정은 스펙에 명시해라.** "WKWebView 는 알아서 될 거야" 같이 가정하지 말고, "이 API 는 이런 함정이 있으니 X 를 채택한다" 로 의사결정을 박아라. 플랫폼 경험이 있는 사람이 스펙을 리뷰하는 게 이상적.

### 3.2 초기 창 크기 누락

**무슨 일이 있었나**: `ContentView` 에 `.frame(minWidth: 600, minHeight: 400)` 만 박혀 있었고, 새 윈도우가 항상 600x400 최소 사이즈로 열렸다. 사용자 피드백 후 `WindowConfigurator` (NSViewRepresentable 에서 NSWindow 직접 조작) 추가.

**스펙에 미리 박았다면**:
```markdown
## 6. UI 구조

### 6.1 윈도우 초기 크기 · 위치
- 첫 열림: 960×720, 스크린 센터
- 사용자 리사이즈 후: 다음 윈도우는 마지막 사이즈·위치 기억 (NSWindow `setFrameAutosaveName`)
- 스크린 크기 대비 최대 90% 로 clamp
```

**교훈**: **UX 기본값을 스펙에서 수치로 박아라.** "적당히" 하지 말고 숫자로.

### 3.3 Open With 경로 리스크 미인지

**무슨 일이 있었나**: 스펙 §5.1 에 "Finder Open With" 를 "지원" 이라고만 적어놨다. 실제로는 SwiftUI `DocumentGroup(viewing:)` + macOS Apple Event odoc 라우팅 이 여러 변수로 얽혀 있어 동작 보장이 안 된다 (CFBundleDocumentTypes, UTI 등록, Launch Services 캐시, LSHandlerRank).

**스펙에 미리 박았다면**:
```markdown
## 5.1 오픈 방식 — 플랫폼 리스크

(a) Finder "Open With"
- 구현: `CFBundleDocumentTypes` + SwiftUI `DocumentGroup(viewing:)` 으로 자동 처리 기대.
```

## 6. 덧붙이는 조언

### 6.1 스펙을 Claude 한테 먼저 쓰게 하지 마라

스펙 자체도 내가 작성은 하지만, **제품 결정은 사람이 먼저 해야 한다**. 내가 스펙을 혼자 쓰면 "Typora 같은 WYSIWYG 무료 뷰어" 같은 포지셔닝까지 추측하는데, 이게 사용자의 진짜 의도와 다를 수 있다.

**권장 워크플로**:
1. 사람이 "이런 제품 만들고 싶다" 한 문단 쓴다
2. Claude 에게 "스펙 문서 템플릿대로 초안을 써줘" 요청
3. 사람이 초안 읽고 수정·보강 (특히 §11 네거티브 규칙)
4. 최종 스펙으로 합의

### 6.2 "돈 지르는 자유" 를 스펙에 박아두면 좋다

이 프로젝트 스펙 §2.3 ("Apple Developer Program 미등록, unsigned DMG"), §2.7 ("UI 영문만, 한글 v2"), §9.3 ("앱 내 업데이트 체크 없음") — 이런 항목들이 **"돈/시간 안 지른다"** 를 명시적으로 박은 것. 내가 "그래도 자동 업데이트는 넣어야…" 하고 고민하지 않고 그냥 skip.

### 6.3 "morning 수동 검증" 섹션을 스펙에 넣어라

이 프로젝트의 `overnight_v0.1.0.md` §3.5 에 "자율 모드에서 검증 불가 → morning 확인 대상" 13 항목이 있었다. 이거 없으면 내가 "완료!" 선언하고 끝낸다. 있으면 RUN_REPORT 에 그 리스트가 복제되고, morning 실사용에서 하나씩 체크 가능.

**표현 예시** (이 프로젝트에서 썼던 그대로):
```markdown
### 자율 모드에서 **검증 불가** → morning 인간 검증 대상
`RUN_REPORT.md` Task 3 섹션 하단에 체크리스트로 명시:
- [ ] 샘플 md 열어서 실제 렌더 품질 (heading/list/code/table/image/link 전부)
- [ ] `⌘+` / `⌘-` / `⌘0` 줌 동작 + 10% step
- [ ] `⇧⌘D` 다크 토글 → CSS 시각 변화
- [ ] 툴바 appearance/줌 아이콘 클릭
...
```

### 6.4 Context 예산을 의식해라

1M context 라도 세션이 길어지면 중간 tool 결과가 쌓여 손해다. **스펙 + 지시서 + RUN_REPORT 를 제대로 유지하면**, 내가 중간 맥락을 까먹어도 이 3 문서만 다시 읽고 복원할 수 있다. 이번 세션은 33 커밋 + 사용자 피드백 루프까지 한 턴에서 돌았는데 그게 가능했던 이유가 이 문서들 덕이다.
